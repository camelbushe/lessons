# Как работает сборка/компиляция
**Внимание** - у меня реализация == определению.
- .h файлы содержат только **объявления**. Их нужно включать туда, где используется указанная в нём функция. Несколько одинаковых **объявлений** не вызовут конфликт.
- Для того чтобы найти реализацию для данной функции нужно указать .cpp файл в котором она содержится.
- Имена .h и .cpp не обязательно должны совпадать.
- На один проект должно быть только одно определение данной функции/идентификатора (Правило одного определения).
- Включать стоит только .h файлы. Так как если в нескольких файлах нужно вызвать функцию, то .cpp приведут к тому, что будет несколько реализаций данной функции, что приведет к ошибке.
- Препроцессор просто вставляет содержимое подключенных файлов прямо в код.


# Приоритет операторов и ассоциативность
Использовать скобки для лучшей читабельности.
```c++
x && y || z;
```
Хоть && имеет приоритет над || лучше заключать в скобки
```c++
(x && y) || z;
```
Заключать в скобки все кроме умножения, деления, сложения, вычитания. Или
```c++
x = (y *= z);
```
В специффикации нет указания в каком порядке вычислять операнды и аргументы функций. Например
```c++
a + b;
```
Нет разницы что первее вычислить a или b. Тоже самое с аргументами функций. Иззаэтого разные компиляторы могут вычислять как справа на лево так и наоборот. Изза этого могут быть разные результаты в зависимости от компилятора.
```c++
#include <iostream>

int getValue() {
  int x{};
  std::cin >> x;
  return x;
}

int sum(int x, int y, int z) {
  return x + (y * z);
}

int main() {
  std::cout << sum(getValue(), getValue(), getValue());
  return 0;
}

```
Изза этого лучше каждое getValue() записать в отдельные переменные а затем их передать в функцию.

(В с++ термин вычисление обычно применяется к вычислению операндов а не всего выражения целиком.)!


# Арифметические операторы
унарный (не бинарный) operator- возвращает результат умноженнный на -1. Аналогично с operator+ но он возвращает это же число.
operator% находит остаток от деления.
operator/ имеет как целочисленное деление так и нет. Если Один или оба операнда это числа с плавающей точкой то результат будет такимже. Если оба целые, то и результат целый.
```c++
2.0 / 4; // Дробное число
2 / 4; // Целое число. (0 в данном случае т.к. целая часть только возвращается)
```
Можно использовать static_cast<> для того чтобы не терять дробную часть при делении целых чисел
```c++
constexpr int x{7};
constexpr int y{2};

std::cout << static_cast<double>(x) / y; // 3.5
```
Деление на 0 приведет к неопределённому поведению. Деление на 0.0 определяется реализацией (компилятором/архитектурой)
|Оператор|Аналог|
|---|---|
|x += y|x = x + y|
|x -= y|x = x - y|
|x *= y|x = x * y|
|x /= y|x = x / y|
|x %= y|x = x % y|
**Модифицирующие операторы** - Возвращают значение и модифицируют один из операндов (=, +=, ...)
**Не модифицирующие операторы** - Только возвращают значение
Операторы можно перегружать и можно сделать чтобы их поведение отличалось от встроенных


# Оператор остатка от деления и возведения в степень
Operator% всегда возвращает число со знаком первого операнда.
Operator% работает как "остаток в школьной математике", а не как математическое modulo
### Возведение в степень
```c++
#include <cmath>
double x{ std::pow(3.0, 4.0) }
```
Параметры и возвращаемое значение имеет тип double


# Операторы инкремента и декркмента. Побочные эффекты
Есть постфиксные и префиксные операторы инкримента и дикримента.
|Версия|Описание|Пример|
|---|---|---|
|Префиксная|Увеличивает/уменьшает значение на 1 и ВОЗВРАЩАЕТ УВЕЛИЧИННОЕ|++x|
|Постфиксная|Увеличивает/уменьшает значение на 1 и ВОЗВРАЩАЕТ ИСХОДНОЕ|x++|
#### Как работает постфиксная
Создается временная копия x с исходным значением. Оригинал увеличивается/уменьшается затем возвращается копия, потом копия удаляется. Эта операция занимает **больше ресурсов**
### Побочные эффекты
Выражение считается имеющим побочный эффект если они делают еще чтото наблюдаемое кроме возвращаения значения. Например x = 5. Тут возвращается 5 но ещё и присваивается значение.

Побочные эффекты + не определенный стандартом порядок вычисления операндов или аргументов функции может привести к разным результатам в зависимости от компилятора
```c++
x + ++x;
```


# Оператор запятая
```c++
(++x, ++y);
```
Сначало выполнится увеличение x, затем увеличение y и вернется увеличенное значение y. Тоесть этот оператор вычисляет сначало левый операнд, затем правый и вернёт правый.
Этот оператор имеет самый низкий приоритет:
```c++
z = a, b;
```
Z получит значение a. b вычислится и отбрасывается.
Изза этого этот оператор лучше вообще не использовать за исключением цикла for.
Не стоит путать этот оператор с разделителем. Напрмер запятая это разделитель аргументов функций.
Не стоит использовать только в качестве разделителя при объявлении нескольких переменнх (т.е. вообще не объявлять так переменные)
```c++
constexpr int x{ 5 }, z{ 6 };
```


# Условный оператор
```text
условие ? выражение1 : выражение2
```
Если условие возвращает true, выполняется выражение1, иначе выражение2.
Этот оператор можно использовать в константных выражениях так как он является выражением в отличии от if...else.
```c++
constexpr bool isTrue{ false };
constexpr int x{ isTrue ? 1 : 0 };
std::cout << x << std::endl;
```
Типы второго и третьего операнда должны либо **совпадать** либо конвертироваться в **одинаковые типы**
```c++
std::cout << (true ? 1 : 2); // Это нориально
std::cout << (true ? 1 ? 2.2); // Это тоже нориально так как 1 можно конвертировать в дробное число
std::cout << (true ? -1 : 2u); // Это тоже сработает, но -1 конвертируется в unsigned int изза этого получается очень большое число. Так не стоит делать
```


# Операторы сравнения. Сравнение чисел с плавающей запятой
Условие в блоке if ожидает булево значение (если не булево, то оно конвертируется). Сравнивать это значение с true или false считается избыточным.
### Больше, меньше с плавающей запятой
Изза ошибок округления (100.0 - 99.9 != 0.1) сравнивать числа с плавающей запятой считается небезопасным, но это может быть приемлимо если в програме не важна такая точность.
### Равно, не равно с плавающей запятой
Использование считается опасным. Когда можно сравнивать
- Сравниваются литералы (**без математических операций**) и число знаков после запятой не привышает максимум для этого типа. Литералы имеют **один и тот же тип**.
```c++
someValue() == 0.1; // Нормально. При условии, что someValue возвращает литерал.
0.7 == 0.7; // Норально.
0.1 + 0.2 == 0.3; // Не нориально. Есть математические операции.
0.9f == 0.9; // Не нормально. Разные типы.
```
### Как можно сравнивать
Можно написать несколько функций
- Использование абсолютной погрешности
```c++
#include <cmath> // Для std::abs

bool absIsEqual(double a, double b, double epsilon) {
    return std::abs(a - b) <= epsilon;
}
```
Сдесь a и b числа, а epsilon это абсолютная погрешность. После какой разничы числа считать идентичными. Минус - нужно выбирать погрешность в зависимости от входных данных.
- Использование относительной погрешности. Это позволяет указывать погрешность в процентах относительно большего числа.
```c++
#include <algorithm> // Для std::max
#include <cmath> // Для std::abs

bool relIsEqual(double a, double b, double epsilon) {
    return std::abs(a - b) <= (std::max(std::abs(a), std::abs(b)) * epsilon);
}
```
Тут погрешность это процент от большего числа. 0.01 = 1%. В этом подходе есть минус - он плохо работает с малыми числами. Например a - 1.0 = 1.11e-16 изза ошибок округления в компьютере. Это максимальное число. Затем возьмём погрешнсть 1e-8. Изза этого будет проходить сравнение 1.11e-16 <= 1.11e-16 * 1e-8. Это вернёт false хотя математически a - 1.0 = 0.
- Использовать комбинированный подход. Мы будем использовать относительную погрешность, но на всякий случай передадим ещё и абсолютную. Важно учесть, что абсолютная погрешность не должна быть меньше чем порог, когда компьтер совершает погрешности при вычислении.

```c++
bool combinedIsEqual(double a, double b, double absEpsilon, double relEpsilon) {
    if (std::abs(a - b) <= absEpsilon) {
        return true;
    } // Если абсолютная погрешность очень маленькая, то это пройдет для маленьких, но не для больших чисел
    return relIsEqual(a, b, relEpsilon);
}
```
### Создание constexpr функций
Эти функции можно сделать constexpr начиная с C++23
```c++
constexpr bool relIsEqual(...) {...};
constexpr bool combinedIsEqual(...) {...};
```
До C++23 функция std::abs не была constexpr поэтому нужно писать свою, чтобы можно было сделать


# Логические операторы.
Operator! имеет довольно высокий приоритет.
### Порядок вычисления операндов
Операторы operator|| и operator&& имеют чёткий порядок вычисления операндов (с лева на право) в отличии от многих других, что является исключением. Например если левый операнд operator&& вернет false, то правый операнд вообще не будет вычислятся.
```c++
(x == 1 && ++y == 2); // ++y не выполнится
```
Но это не работает с перегруженными операторами. Они не будут так себя вести.
### Законы де Моргана
```c++
!(x && y) == (!x || !y);
!(x || y) == (!x && !y);
```
### Логический оператор XOR
Это так называемое строгое ИЛИ. Возвращает true только если оба операнда разные но false даже если оба true. Этот оператор отсутствует в языке. Но operator!= по факту выполняет ту же функцию.
```c++
!!a != !!b != !!c; // Так можно работать если a, b, c не булевы
```
### Альтернативное обозначение
Данные операторы можно записать как or and not, но предпочтительнее использовать символы вместо текста.


# Блоки
Также называют состовное операторное выражение. Не требуют после себя точку с запятой. Могут вкладыватся друг друга. Максимальный уровень вложенности согласно стандарту 256, но некоторые могут поддерживать меньше. Уровень влоденности желательно не должен превышать 3.


# Пространства имён
Позволяют избегать появления одинаковых имёе идентификаторов так как каждый новый идентификатор можно помещать в собственное пространство имён.
Название желательно писать с большой буквы, но можно и с маленькой. Пространство имён может быть объявлено только в глобальной оьласти видимости или внутри другого пространства имён.
```c++
namespace Plus {
    int doSomething(int a, int b) {
        return a - b;
    }
} // Объявление пространства имён

Plus::doSomething(1, 2); // Вызов функции из пространства имён

::doSomething(1, 1); // Валидная запись, но это ищет doSomething в глобальной области видимости. Это тоже самое если написать doSomething без оператора разрешения области видимости.
```
Использовать ператор разрешения области видимости без указания имени пространства имён полезно чтобы если функция вызывается внутри пространства имён, компилятор **сразу использовал глобальное пространство имён**, а не искал поднимаясь вверх.
```c++
void print() {
    std::cout << "Hello";
}

namespace Foo {
    void print() {
        std::cout << "Hi";
    }
    void greet() {
        ::print(); // Из глобального пространства имён
        print(); // Из текущего пространства имён
    }
}
```
### Предварительные определения
Нужно также писать в прострпнствах имён по такойже структуре.
```c++
namespace Plus {
    int doSomething(int a, int b);
}
```
Если будет несколько объявлений одного и того же пространства имён как в нескольких файлах так и в одном, то это не приведёт к ошибке. Они просто объединятся.
```c++
namespace Greet {
    void sayHi() {
        std::cout << "Hi";
    }
}

namespace Greet {
    void sayHello() {
        std::cout << "Hello";
    }
}
```
Не стоит расширять таким образом пространство имён std.
### Вложенные пространства имён
```c++
namespace Foo::Bar {
    void sayHi() {
        std::cout << "Hi";
    }
} // Это эвивалентно примеру ниже

namespace Foo {
    namespace Bar {
        void sayHi() {
            std::cout << "Hi";
        }
    }
}

Foo::Bar::sayHi(); // Вызов из вложенного пространства имён
```
Так тоже приемлимо
```c++
namespace Foo::Bar {
    void sayHi() {
        std::cout << "Hi";
    }
}

namespace Foo {
    void sayHello() {
        std::cout << "Hello";
    }
} // Добавляет только в Foo
```
### Псевдонимы пространства имён
Можно добавить псевдоним для пространства имён и к пространству можно будет обращаться как через псевдоним так и через изначальное имя
```c++
namespace Greet = Foo::Bar;
```
Следует избегать вложенности в пространствах имён более 3 уровней


# Локальные переменные
Локальные переменные имеют блочную область видимости. Нельзя получить доступ из родителя, но можно из дочерних. Они создаются в момент определения и уничтожаются в конце блока в котором они определены. 

Если в родителе есть переменная с именем как в текущем блоке, то будет использоваться из текущего блока (Скрытие имени, затенение). Но если перекрывается глобальная переменная, то к ней можно получить доступ через operator::. Следует избегать переопределения глобальных и локальных переменных.

Не стоит создавать блоки, единственная цель которых - ограничение области видимости. Переменную не стоит размещать во внешнем блоке если она нужна только в текущем.

# Введение в глобальные переменные
Объявляются вне любых функций (После директивы include) или в пространстве имён (предпочтитеьнее). Они видны в любом месте програмы. Имеют статическую/длительность, тоесть создаются при запуске програмы до началы выполнения main и уничтожаются при ее завершении.

Стоит добавлять префикс g или g_ если глобальные переменные одъявлены в **глобальном** пространстве имён.

Глобальные переменные по умолчани инициализируются нулём
```c++
#include <iostream>

int a; // Содержит 0

int main() {
    int b; // Не содержит 0
    return 0;
}
```
Но const и constexpr все равно должны иметь инициализатор даже если они глобальные.
```c++
#include <iostream>

const int a; // Ошибка
const int b{}; // Нориально ({} инициализирует нулём)
constexpr int c = 0; // Нормально
```
Использование неконстантных глобальных переменных следует избегать. 


# Внутренняя свезь
В языке есть 2 типа связи: внутреняя и внешняя.
- **Внешняя** - можно получить доступ из разных файлов (единиц трансляции). Например я объявляю глобальную переменную a в разных файлах. Это приведёт к ошибке изза повторного определения.
```c++
// my_lib.cpp
int a{ 10 };
```
```c++
// main.cpp
#include <iostream>

int a{ 10 };

int main() {
    std::cout << "Hello world!";
}
```
- **Внутреняя связь* - это значит, что переменная видна только в текущем файле (единице трансляции) и повторное определение точно такой же переменной в другом файле не приведёт к ошибке.

Как указать внутреннюю связь.
- Её по умолчанию имеют const и constexpr переменные.
- С помощью ключевого слова static
```c++
static int a{ 10 }; // Видна только в указанном файле
```
### Когда это использовать
- Идентификатор не предназначен для использования в других файлах
- Запрещено использовать идентификатор в других файлах (**Рекомендует руководство**)
### Почему const и constexpr по умолчанию имеют внутреннюю связь.
- Константные переменные нужны просто для инициализации какихто занчений. Они подключаются через заголовки в разные файлы только для того чтобы не прописывать каждый раз их определение.
- Обычные переменные подключаются через заголовочные файлы для того, чтобы взаимодействовать с разными частями програм. Например для изменения счёта игрока из разных файлов.

Если бы они имели внутреннюю свяь, то это бы приводило к ошибкам повторного определения **при подключении к друним .cpp файлам**. Так как в двух файлах было бы одно и тоже определение.
Но внутреняя связь предотвращает это так как в каждый файл вставляется объявление с внутренней связью.


# Внешняя связь
Идентификаторы с внешней связью видны в других файлах
### Как указать внешнюю связь
- Функции и глобальные переменные имеют по умолчанию
- С помощью ключевого слова extern
```c++
extern const int a{ 10 }; // Константа будет видна в других файлах, но чтобы ее использовать, нужно выполнить предварительное объявление в других файлах
```
### Предварительное объявление глобальных переменных
Если переменная инициализирована в другом .cpp файле, а **не в .h**, то ее нужно предварительно объявить c помощью extern
```c++
// util.cpp
extern const int a{ 10 }; // extern служит для указания внешней связи и это слово не нужно при инициализации для не константных переменных
```
```c++
// main.cpp
#include <iostream>

extern const int a; // extern говорит, что переменная a внешняя. Тоесть мы не создаём новую переменную a, котора будет иметь внутреннюю связь в этом файле не смотря на то, что в другом файле у нее есть extern

int main() {
    std::cout << a;
}
```
Следует сравнить с этой записью:
```c++
// util.cpp
extern const int a{ 10 };
```
```c++
// main.cpp
#include <iostream>

const int a; // Привело бы к ошибке, так как мы не указали, что используем переменную a, которая объявлена гдето (не используем предварительное объявление) и компилятор думает, что мы создаём новую переменную (она будет иметь внутреннюю свзяь и не приведёт к ошибке нескольких определений) и не даём ей значение (константы требуют инициализатор).

int main() {
    std::cout << a;
}
```
Или сравнение с обычными переменными
```c++
// util.cpp
int a{ 10 }; // Не пишу extern так как эта переменная уже имеет внешнюю связь
```
```c++
// main.cpp
#include <iostream>

int a; // Повторное определение так как a в этом файле имеет сразу внешнюю связь
static int a; // Ошибки нет. Просто создаём новую переменную. Она имеет внутреннюю связь и не связывается с переменной из другого файла
extern int a; // Ошибки нет. Говорим, что это НЕ новое определение, а предварительное объявление

int main() {
    std::cout << a;
}
```
Для предварительного объявления функции слово extern не требуется.
Переменные с constexpr нельзя определять предварительно так как такое выражение требует инициализатора, а так как каждый файл компилируется по отдельности это приведет к ошибкам:
```c++
// util.cpp
extern constexpr int a{ 10 };
```
```c++
// main.cpp
#include <iostream>

extern constexpr int a; // Ошибка
extren int a; // Ошибка
extern const int a; // Сработает и значение будет 10
```


# Почему НЕКОНСТАНТНЫЕ глобальные переменные это зло
- Их может изменить любая функция. 
- Это усложняет отладку так как нужно пересматривать всю программу.
- Переменные из разных единиц трансляции могут инициализироваться в разном порядке. Например сначало могут инициализироваться переменные из logger.cpp, а потом из database.cpp, а может и наоборот
## Когда их стоит использовать
Можно использовать, если глобальная переменная соответствует как минимум этим двум критериям:
- Она хранит объект, который может быть только один во всей програме (Глобальный логгер, генератор случайных числел, конфигурация приложения)
- Этот объект используется практически в любом файле програмы
## Правильное использование глобальных переменных
Это применимо и к константным переменным
- Добавлять префикс g или g_ к переменным без пространста имён
- Поместить в отдельное пространство имён
- Инкапсулировать переменную (сделать ее доступной только из файла, в котором она объявлена) и предоставить функцию для работы с ней:
```c++
namespace Constants {
    constexpr duble gravity{ 9.8 };
}
double getGravity() {
    return Constants::gravity;
}
```
- Если функция использует глобальную переменную, нужно сделать так, чтобы функция не на прямую ее использовала, а получала через аргументы:
```c++
doube veliocity(int time, double gravity) {
    return time * gravity;
} // Использовать это

double veliocity(int time) {
    return time * Constants::gravity;
} // Вместо этого
```

# Встраеваемые функции и переменные
Предположим у нас есть такой код.
```c++
int sum(int a, int b) {
    return a + b;
}

int main() {
    std::cout << "2 + 2 = " << std::cout << sum(2, 2) << std::endl;
    std::cout << "3 + 3 = " << std::cout << sum(3, 3) << std::endl;

    return 0;
}
```
Тут есть недостатки. Каждый вызов функции незначительно снижает производительность. Так как процессору нужно:
- Запомнить адрес текущей выполняемой инструкции
- Создать и инициализировать параметры функций
- И так далее...
## Как это оптимизирует компилятор
Компилятор может выполнить встраивание кода. Это процесс, когда вызов функции заменяется кодом из её определения. Это будет выглядеть так:
```c++
int main() {
    std::cout << "2 + 2 = " << std::cout << 2 + 2 << std::endl;
    std::cout << "3 + 3 = " << std::cout << 3 + 3 << std::endl;

    return 0;
}
```
Это позволяет избежать не только выше перечисленных проблем, а ещё так как это теперь константное выражение 2 + 2 и 3 + 3 могут теперь вычислиться на этапе компиляции.
Компилятор обычно решает нужно ли так делать, так как в некоторых случаях это может навредить. Например каомпилятор может это не делать и изза того, что определение функции находится в другом файле.
## Ручнаое встраивание кода
Раньше компиляторы могли плохо определять нужно ли выполнять встраивание кода. Изза этого было добавлено ключевое слово inline. Оно позволяет подсказать компилятору, что возможно встраивание кода может помоч оптимизации. Компилятор в праве это проигнорировать
```c++
inline int sum(int a, int b) {
    return a + b;
}

int main() {
    std::cout << "2 + 2 = " << std::cout << sum(2, 2) << std::endl;
    std::cout << "3 + 3 = " << std::cout << sum(3, 3) << std::endl;

    return 0;
}
```
Сейчас использование этого слова считается плохим тоном. Так как это было всего лишь подсказкой и имело ещё недостатки в использовании.
## Современное использование ключевого слова inline
Данное слово позволет иметь несколько определений в разных единицах трансляции. Более того если у меня есть несколько таких определений, то память расходуется так как будто это одно определение. Это значит я могу определит функцию sum и в файле utils.cpp и в файле main.cpp
```c++
// utils.cpp
inline int sum(int a, int b) {
    return a + b;
}
```
```c++
#include <iostream>

inline int sum(int a, int b) {
    return a + b;
}

int main() {
    std::cout << sum(3, 3) << std::endl;
}
```
Это также позволяет безопасно писать определение функций в заголовочном файле.
### Требование к таким функциям
- Компилятор должен видеть полное определение функции в каждой еденице трансляции. Предварительного объявления не достаточно.
```c++
#include <iostream>

inline int sum(int a, int b); // Даже если определение есть в другом файле, использование этой функции в данном файле приведёт к ошибке так как нет определения.

int main() {
    std::cout << sum(3, 3) << std::endl;
}
```
- В каждой единицы трансляции функцию можно определять только один раз
```c++
#include <iostream>

inline int sum(int a, int b) {
    return a + b;
}

inline int sum(int a, int b) {
    return a + b;
} // Ошибка

int main() {
    std::cout << sum(3, 3) << std::endl;
}
```
- Определение может быть после использования, но должно все равно быть предварительное объявление. Однако в таком случае компилятор скорее всего не выполнит встраивание кода пока не увидит определение. Только выполнит встраивание кода в местах, которые идут после определения.
- Если функция имеет внешнюю связь, то ее определения должны быть идентичными иначе будет неопределённое поведение.
Следует избегать использования этого слова без веской причины (если определяю функции и переменные в заголовочных файлах).
Некоторые функции могут быть встраиваемыми по умолчанию. Например constexpr и consteval функции.
## Встраиваемые переменные
В C++17 появились встраиваемые переменные. Позволяют иметь несколько определений.
### Требования
- Компилятор должен видеть полное объявление в каждой единице трансляции
- В каждой единицы трансляции можно определять только один раз
Существуют переменные, которые являются встраиваемыми по умолчанию


# Способы использования глобальных констант в нескольких файлах
## Определение в заголовочном файле
```c++
// constants.h
namespace Constants {
    constexpr double pi{ 3.14 };
    constexpr double avogadro{ 6.02e23 };
}
``` 
```c++
// main.cpp
#include <iostream>
#include "constants.h"

int main() {
    std::cout << Constants::pi << std::endl;
}
```
### Недостатки данного подхода
- Каждый раз вместо #include "constants.h" вставляются определения констант. Это может нагружать память если они весят много.
- Изменение одной константы требует перекомпиляции каждого файла с #include "constants.h"
### Преимущества
- Использование до C++17
- Константы могут использоваться в постоянных выражениях
## Определение в .cpp и предварительное объявление через .h
```c++
// constants.cpp
namespace Constants {
    extern constexpr double pi{ 3.14 };
    extern constexpr double avogadro{ 6.02e23 };
} // Объявляем и устанавливаем внешнюю связь
``` 
```c++
// constants.h
namespace Constants {
    extern const double pi;
    extern const double avogadro;
} 
// Используем extern для "импорта" переменной (указываем что используем переменную с глобальной связью)
// Использование const вместо constexpr не приведёт к ошибке. Иначе програма не скомпилируется, но переменные не смогут использоваться в константных выражениях
``` 
```c++
// main.cpp
#include <iostream>
#include "constants.h"

int main() {
    std::cout << Constants::pi << std::endl;
}
```
### Недостатки данного подхода
- Переменные нельзя использовать в константных выражениях
### Приимущества
- Использование до C++17
- Есть только одна копия переменной, что экономит память
## Использовать inline в .h файлах
```c++
// constants.h
namespace Constants {
    inline constexpr double pi{ 3.14 };
    inline constexpr double avogadro{ 6.02e23 };
} // inline позволяет сделать так, что память будет использоваться как у одного объявления даже если будет много #include "constants.h" 
``` 
```c++
// main.cpp
#include <iostream>
#include "constants.h"

int main() {
    std::cout << Constants::pi << std::endl;
}
```
### Недостатки
- Изменение одной константы требует перекомпиляции каждого файла с #include "constants.h"
- Не работает до C++17
### Приимущества
- Экономия памяти
- Можно использовать в константных выражениях


# Статические локальные переменные
Клчевое слово static с локальными переменными даёт им статическую длительность. Это значит, что переменная создаётся в начале програмы и уничтожается в конце (как глобальная переменная). Это значит, что переменная сохранит свое значение при выходе из области видимости. Переменная продолжает иметь блочную область видимости.
```c++
#include <iostream>

void increment() {
    int a{ 0 };
    std::cout << a;
    ++a;

}

int main() {
    increment();
    increment();
    increment();
    
    return 0;
}
// В консоли будет 000.
```
```c++
#include <iostream>

void increment() {
    static int a{ 0 }; // Переменная статическая
    std::cout << a;
    ++a;

}

int main() {
    increment();
    increment();
    increment();
    
    return 0;
}
// В консоли будет 012
```
### Инициализация статических локальных переменных
Если нет начального значения или переменная имеет non-constexpr инициализатор, то она инициализируется нулями на старте програмы. Когда выполнее доходит до ее первого определеня они повторно инициализируются указанным значением. Определение пропускаются при последующих вызовах.

Если определение через {} (без значения в скобках) или если инициализатор constexpr, инициализация происходит при запуске при запуске.
## Добавления префикса
Статическим локальным переменным следует добавлять префикс s_.


# Повторение терменологии
## Область видимости
Это откуда можно получить доступ к идентификатору
### Локальная (блочная) область видимости
Доступны с момента объявления и до конца блока.
- Параметры функции
- Локальные переменные
### Глобальная область видимости
Доступны с момента объявления и до конца файла
- Глобальные переменные
- Функции
## Длительность
Когда переменная создаётся и уничтожается
### Автоматическая
Создаются в момент определения и уничтожаются при выходе из блока
- Параметры функции
- Локальные переменные
### Статическая
Создаются в момент запуска програмы и уничтожаются при завершении
- Глобальные переменные
- Статические локальные переменные
### Динамическая
Создаются и уничтожаются по запросу програмиста
- Динамически выделяемые переменные
## Связь
Будет ли объявление того же идентификатора в другой области видимости ссылаться на ту же сущность
### Не имеют связи
Каждое объявление ссылкается на уникальную сущность
- Локальные переменные
### Внутренняя связь
Объявление идентификатора **в одной и той же единице транслции** ссылается на одну и ту же сущность
- Статические глобальные переменные
- Статические функции
- Константные глобальные переменные
### Внешняя связь
Объявления одного и того же идентификатора **во всей программе** ссылаются на одну и ту же сущность
- Не статические функции
- Неконстантные глобальные переменные
- Внешние константные глобальные переменные
- Инлайновые константные глобальные переменные
- Пространства имён

Тут перечислены только то, что было изучено до этой главы
## Предварительное определение
Если нужно получить переменную и функцию из другого файла можно использовать предварительное объявление.
```c++
void increment(); // Предварительное определение функции
extern int a; // Предварительное определение глобальной переменной
extern const int a; // Предварительное определение константы. Может быть использовано для предварительного определения constexpr, но оно не будет constexpr в данной единице трансляции
extern constexpr int a; // Нельзя
```
Предварительно объявленные переменные имеют обычную область видимости:
```c++
int main() {
    {
        extern int a;
        std::cout << a;
    }

    std::cout << a; // Ошибка т.к. переменная в другом блоке при том что a в другом файле является глобальной
    return 0;
}
```
Следущий код также работает нормально:
```c++
#include <iostream>

extern int b;

int main() {

    std::cout << b; // 5
    return 0;
}

int b{ 5 };
```
## Спецификатор класса хранения
При использовании ключевых слов extern и static в качестве части объявления идентификатора их называют **спецификатором класса хранения**. В языке есть ещё несколько: mutable, auto, register, thread_local


# Using декларации и директивы
## Квалифицированные (полные) и неквалифецированные (неполные) имена
**Квалифицированное имя** - имя, содержащее квалификатор области видимости:
```c++
std::cout
::foo

Some_class::s_member
obj.x
ptr->y 
```
**Неквалифицированное имя** - не содержит квалификатора области видимости. Например просто cout или x
## Using декларации
Оператор using позволяет использовать неполное имя в качестве псевдонима для полного:
```c++
int main() {
    using std::cout; // Позволяет писать просто cout вместо std::cout. Но только для std::cout, а не всего std
    cout << "Hello world!" << std::endl;
}
```
Если возникнет конфликт между cout из std и другим cout. Предпочнение будет отдано тому, что из std в данном случае.
## Using директивы
Позволяет писать неполные имена для всего пространства имён
```c++
int main() {
    using namespace std;
    cout << "Hello world!" << endl;
}
```
### Почему следует избегать using namespace
- Включают имена, которые могут не использоваться в програме
- Не отдаёт предпочтение при конфликте указанному пространству имён
```c++
int cout() {
    return 5;
}

int main() {
    using namespace std;
    cout << "Hello world!" << endl; // Ошибка. Компилятор не знает какой cout использовать.
}
```
## Облась видимости слова using
Не действует в родительских или соседних блоках. Если написано в глобальной области видимости, то видно только в текущем файле.
## Не стоит использовать using там, где он может повлиять на код в других файлах
Например в заголовочных файлах или перед директивой include. И даже внутри функций в заголовочных файлах (это тоже имеет последствия):
```c++
using std::cout;
#include "utils.h" // Так как компилятор вставляет код из utils.h это повлияет на код в других файлах.
```
